Netty 启动时默认的 BOSS、IO 组线程数？

boss 线程即为监听线程，负责处理客户端的 TCP 连接请求。如系统只有一个服务端端口需要监听，则建议 bossGroup 线程组线程数设置为1。

io 线程即为工作线程，负责 io 操作，调用用户的方法。当消息被 BOSS 线程移交给 IO 线程后，从消息的读取、编码以及后续 Handler 的执行都由该线程服务，整个流程不会进行上下文切换

EventLoopGroup 为 EventLoop 线程组，如果没有指定线程个数，默认取 -Dio.netty.eventLoopThreads（MultithreadEventLoopGroup），如果该系统参数也没有指定，则为可用的CPU内核数 × 2

消息写出会过 Handler 吗？分析 ChannelHandler 类继承结构

消息是从 Handler 流出的。
ChannelPipeline 通过 ChannelHandler 接口来实现事件的拦截和处理，由于 ChannelHandler 中的事件种类繁多，不同的 ChannelHandler 可能只需要关心其中的某一个或者几个事件，所以，通常 ChannelHandler 只需要继承 ChannelHandlerAdapter 类覆盖自己关心的方法即可。

一个连接中只有一个相应的实例？每来一个消息都会生成一个相应的实例？又或者整个系统共享一个实例？
一个 Channel 具有多个 Handler 实例，且每个 Channel 都存有一份实例。当然 Handler 也可以不同的 Channel 之间共享
一个 Channel 对应一个 ChannelPipeline
一个 Handler 对应一个 ChannelHanlderContext，且每次事件都用的是同一个 ChannelHanlderContext，只要当前连接还保持

Netty 中发送消息的两种方法

当一个 ChannelHandler 添加到 ChannelPipeline 中时获得一个ChannelHandlerContext。通常是安全的获得这个对象的引用，但是当一个数据报协议如UDP时这是不正确的，这个对象可以在之后用来获取底层通道，因为要用它来read/write消息，因此通道会保留。也就是说Netty中发送消息有两种方法：直接写入通道或写入 ChannelHandlerContext 对象。这两种方法的主要区别如下：

  • 直接写入通道导致处理消息从ChannelPipeline的尾部开始
  • 写入 ChannelHandlerContext 对象导致处理消息从 ChannelPipeline 的下一个handler开始

在 InboundHandler01 中使用 Channel 直接写入，得到的效果为在 OutboundHandler01 中打印消息；使用 ChannelHandlerContext 写入，得到的效果为没经过 OutboundHandler01，但是消息还是写到客户端。

而在 Server 端未使用 SimpleChannelInboundHandler 的原因是我们希望 Server 端能够将接收到的消息写回给 clients，也就是说，由于写操作可能在 channelRead 返回后完成（写操作是异步的），所以我们不能在 channelRead 中释放资源，当写操作完成后，netty 会自动释放消息。

为什么要在父类的泛型限定中指定的是自己的子类

如类 AbstractBootstrap 的定义如下：

指定 B 类型为当前类的子类，C 类型为 Channel 类的子类，为什么要如此设计？

经观察发现 AbstractBootstrap 类的实现类有两个：ServerBootstrap 和 Bootstrap，为服务器和客户端的启动引导帮助类。B 类型均作为方法的返回值类型，再观察 B 的具体类型，推断可知如此设计，只需在父类中编写一套模板代码，子类便可调用父类的方法实现链式编程。在创建类的时候，让父类知道当前的子类的类型，泛型！再用该子类的类型作为返回值，去掉重复的代码，实现链式编程。
我的实现方式：

也可以实现链式编程，虽然说返回的不是一个具体的类型，但是外界（）为什么需要知道具体的类型？父类也可以调用！这是弊端，如果你返回的是父类的类型，就无法调用具体子类的方法，即使当前是子类的实例。

此实现只能返回的是父类的引用，父类不知道子类的存在，所以当外界需要子类引用时，就无法满足需求。在 Netty 实现中，当子类（如：ServerBootstrap）调用的 channel() 时，使用的是父类的方法，但是父类通过泛型限定能知道具体的子类类型，返回的是具体的子类引用，而不是父类的引用。
