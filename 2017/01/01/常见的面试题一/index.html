    • 1. obj.hashcode()、obj.equals(b)

    作用

    hashcode() 方法用来获取对象的哈希值，提高哈希表的性能（如 HashMap、HashTable）

    hashcode 工作原理

    hashcode 是用来查找的，假如内存中有依次八个位置

    现有个类，类中有个字段为 id，需要把这个类放在8个位置之一。如果不用 hashcode 而任意存放，当查找时候，则需要对这8各个位置进行遍历，或者使用二分法之类的算法。
    如果使用 hashcode 效率将会大大的提高。
    定义 hashcode 的值为 id%8，取得余数则为存放的位置。如：假设 id=9，9%8余数为1，于是存放的位置为1。假设 id=13，13%8余数为5，于是存放的位置为5。于是，每次需要查找的时候只需要通过 id%8 这个算法求余数，直接找到存放的位置了。

    与 equals() 之间的联系

    当两个类的 hashcode 值一样时候（碰撞），则需要 equals() 来帮忙。如：9%8和17%8的余数都是1。这里可能可能会想到既然键的 hashcode 都一样了，为什么还能存入？答案是：能存入，只要满足重写 equals() 这个前提要求。也就是说，总得让虚拟机区分的出来，对象与对象之间的区别。即：当对象的 hashcode 值相等时，只要不满足 a.equals(b) 为 true 时，对象就能存入！同样的道理，当取出时，先比较对象的 hashcode 值，再去调用 equals()。

    先通过 hashcode来判断两个类是否存放某个桶里，但这个桶里可能有很多类，那么就需要再通过 equals 来在这个桶里找到我们要的类。

    2. final、finalize、finally

    final 是 java 关键字。 当一个变量声明为 final 时，只被初始化一次，且不能被改变。当一个类声明为 final 时，则该类不能被继承。当一个方法被声明为 final 时，则该方法不能被重写。

    finally 是代码块。 当存在 try 语句存在时，finally 代码块中的语句总能被执行。即使是意外的异常的发生，finally 代码块中的语句也能被执行。除处理异常情况之外，还避免了开发者无意间因为 return,continue,break 等语句，跳过了关闭清理资源的动作的情况的发生。把关闭清理资源的动作放在 finally 语句块中无疑一个好的实践。即使是代码没有事先预料到的异常发生。

    finalize 是方法。 在对象被回收之前，会先调用此方法。可以在此方法里面添加添加自己的回收资源的代码。

    3. String、StringBuffer 、StringBuilder

    String 是不可变对象（一旦创建无法再被改变），以字符串的形似被创建以后，存放于字符串常量池。
    在 java 中，不可变对象都是线程安全的，也暗示着 String 对象也是线程安全的。String 对象不能同时被两个线程同时使用。

    StringBuffer 是可变对象，意味着可以改变这个对象的值。该类被创建在 Java 堆中。拥有和 StringBuilder 一样的方法，不同的是 StringBuffer 中的每个方法都是线程同步的，即 StringBuuffer 是线程安全的。
    因为不允许多个线程同时访问同一个方法，在同一时间该类方法只能由一个线程执行。由于线程安全的特性，性能不足是 StringBuffer 的缺点。同样的方法 StringBuffer 的效率比 StringBuilder 要低。

    StringBuilder 和 StringBuffer 一样，存储在堆中也可以被修改。最大的区别就是 StringBuilder 是线程不安全的，而StringBuffer 是线程安全的。StringBuilder 性能较快，线程不安全

    String、StringBuffer、StringBuilder 区别表

    属性 / 字段 String StringBuffer StringBuilder
    存储位置 常量池 堆 堆
    是否可修改 否 是 是
    线程安全 是 是 否
    性能 快 慢 快

    4. HashMap、HashTable、TreeMap、ConcurrentHashMap

    HashMap 是基于哈希表的 Map 接口的实现，本质是数组加链表。根据 key 取得hash 值，然后计算出数组下标，如果多个 key 对应到同一个下标，就用链表串起来，新插入的在前面。特性如下：

    • 允许使用 null 键和 null 值
    • 不保证映射的顺序，特别是不保证该顺序恒久不变
    • 线程不安全

    若想使用线程安全的 HashMap，可使用

    或者是 ConcurrentHashMap

    HashTable 是遗留类，功能与 Hashmap 基本相似，不同的是 HashTable 继承自 Dictionary 类，并且是线程安全的。任意的时间只能有一个线程能写 HashTable，但是并发性不如 ConcurrentHashMap，因为 ConcurrentHashMap 引入了分段锁，而 HashTable 用的是全局锁。

    TreeMap 实现 SortedMap 接口，能够将保存的记录排序，默认是键的升序，也可以指定比较器。当 Iterator 遍历 TreeMap 时，得到的记录是排过序的。在使用TreeMap时，key必须实现Comparable接口或者在构造TreeMap传入自定义的Comparator，否则会在运行时抛出java.lang.ClassCastException类型的异常。

    ConcurrentHashMap 是并发散列表的实现，允许完全并发的读取，支持给定数量的并发更新。相比同步包装器包装的 HashMap（Collections.synchronizedMap(new HashMap())），ConcurrentHashMap 拥有更高的并发性。来源如下：

    • 用分离锁实现多个线程间更深层次的共享访问
    • 用 HashEntry 对象的不变性来降低执行读操作的线程在遍历链表期间对加锁的要求
    • 通过对同一个 Volatile 变量的写 / 读访问，协调不同线程间的内存可见性

    5. 在线程安全的前提下实现一个单例模式

    问题：
    当多个线程同时调用 getInstance() 方法，就有可能会发生多个线程同时通过条件检查，此时就会有多个实例。

    问题：
    此版本只是将 new Singleton() 这个操作串行化，并没有保证单实例。因为多线程还是有可能在同时通过条件检查。

    问题：
    此版本保证了，创建对象的动作只有一次。后面的动作全是读取那个成员变量，这些读取的动作不需要线程同步，浪费性能。

    问题：
    这个版本又叫 “双重检查”Double-Check。如果实例创建了，那就不需要同步了，直接返回就好了。不然，就开始同步线程。如果被同步的线程中，有一个线程创建了对象，那么别的线程就不用再创建了。
    问题在于singleton = new Singleton()这句，这并非是一个原子操作，事实上在 JVM 中这句话大概做了下面 3 件事情。

    1. 给 singleton 分配内存
    2. 调用 Singleton 的构造函数来初始化成员变量，形成实例
    3. 将singleton对象指向分配的内存空间（执行完这步 singleton才是非 null 了）

    但是在 JVM 的即时编译器中存在指令重排序的优化。也就是说上面的第二步和第三步的顺序是不能保证的，最终的执行顺序可能是 1-2-3 也可能是 1-3-2。如果是后者，则在 3 执行完毕、2 未执行之前，被线程二抢占了，这时 instance 已经是非 null 了（但却没有初始化），所以线程二会直接返回 instance，然后使用的话有可能会报错。

    问题：
    当这个类被加载的时候，new Singleton() 这句话就会被执行，就算是getInstance()没有被调用，类也被初始化了。

    当类的构造函数中，有一些事可能需要依赖于别的类干的一些事（比如某个配置文件，或是某个被其它类创建的资源），希望他能在我第一次 getInstance() 时才被真正的创建，而不是把类的创建委托给了类装载器。

    问题：
    上面这种方式，仍然使用JVM本身机制保证了线程安全问题；由于 SingletonHolder 是私有的，除了 getInstance() 之外没有办法访问它，因此它只有在getInstance()被调用时才会真正创建；同时读取实例的时候不会进行同步，没有性能缺陷；也不依赖 JDK 版本。
